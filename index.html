The things I do

I am a tech lead with ThoughtWorks. My sweet spot is leading small teams working on small systems. My view is that most systems could be smaller and should have less complexity. My approach is to strip away superfluous details and build only what is needed now to meet the proven business need.

My recent experience has been in JavaScript including mobile and web projects, node services and lambdas. Prior lives have been lived near Ruby, Java and C++.

I'm a polyfill. In 2010 I joined ThoughtWorks as a tester with a passion for software testing. Before that, I was a developer. During my years at ThoughtWorks, I've added more strings to my bow: some analysis work, some experience design work before going back to software development.

GraphCI

The docker-first and docker-last graph based CI system with package support.

Fed up with the plethora of second-rate CI systems currently in the market, a friend and I built our own CI system. Our core difference was to formalise docker into the system as docker is added onto almost every build system I have seen in the past five years. This also allows the user to run the exact same build locally and remotely.

The second aspect is an immutable system to ensure that one build step cannot modify the results of completed steps. This is common in build systems and makes diagnosing build failures more difficult than they need to be.

The third aspect was the concept of a build package system. This would allow users to define common deployment systems that can be shared with other users. I need to deploy some node lambdas to AWS. There is a package for that. I need to build a Jar. There is a package for that.

I've worked on a dozen different build systems and I find myself and everyone else repeating the same thing again and again. With packages, we could start leveraging that work the same way that we do with packages in modern programming languages.

GraphCI is open-source, functional and no longer in active development.

Ensemble

Multi-device, multiplayer game engine

Multiplayer gaming engines are notoriously hard to write, distributed ones even more so. The challenge of supporting low latency, high bandwidth behaviour over a network that is neither stops small teams from trying.

Cross-platform is the scourge of web-developer around the world. Innumerable device sizes and resolutions, a scattergun approach to standards implementation and performance quirks.

Sensing an opportunity I set out with a friend to build a game engine to support small game development shops in overcoming both challenges. Ensemble is the result.

Ensemble is open-source, functional and no longer in active development.

An Insurance Company [again]

2.5 months, Melbourne

I ventured back into this insurance company to coach a pair of their graduates on building out a replacement test tool. The client had an application that collected SOAP XML documents from a queue and stored them in a database. A server-side web application would allow filtering over the dataset. It suffered performance problems

We replaced it with an AngularJS client (shifting the HTML construction performance to each client), a service that serves JSON documents (with hypermedia controls). The queue reader became a standalone app. This allowed us to dedicate more instances to the queues that get more activity.

The final part was profiling the components and the DB and making tweaks to improve throughput (on the queue reader) and reduce latency (on the website). The application supports a hundred users and processes between 200,000 and 500,000 messages per day.

javascript, angular
